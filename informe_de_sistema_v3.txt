1. RESUMEN EJECUTIVO
================================================================================

Delivery360 es un sistema enterprise de gestion de entregas con arquitectura
moderna basada en microservicios. El proyecto utiliza FastAPI (Python) para
el backend y Next.js (React/TypeScript) para el frontend.

ESTADO ACTUAL (VERSION 3.0):
- Backend: 100% COMPLETADO
  * Todos los modelos de datos implementados (13 modelos principales)
  * APIs REST completas para todos los recursos
  * Autenticacion JWT funcional con access/refresh tokens
  * Sistema de autorizacion por roles (superadmin, gerente, operador, repartidor)
  * Migraciones de base de datos configuradas con Alembic
  * Tareas background con Celery
  * Monitoreo con Prometheus/Grafana
  * Rate limiting implementado
  * Auditoria completa de acciones (LGPD compliant)
  * WebSockets configurados para tiempo real
  
- Frontend: En desarrollo (FASE 2 pendiente)
  * Scaffold base creado con Next.js
  * Componentes placeholder implementados
  * Pendiente: integracion con APIs del backend
  
- Infraestructura: 100% configurada
  * Docker y Docker Compose para desarrollo
  * Kubernetes para produccion
  * CI/CD pipelines
  * Monitoreo y logging centralizado

TECNOLOGIAS PRINCIPALES:
Backend: Python 3.11, FastAPI 0.111.0, SQLAlchemy 2.0, PostgreSQL 16, Redis 7
Frontend: Next.js latest, React latest, TypeScript, Tailwind CSS
Infraestructura: Docker, Kubernetes, Prometheus, Grafana, Sentry, ELK Stack

ARCHIVOS PRINCIPALES DEL BACKEND:
- Total de archivos Python: 106+ archivos
- Modelos de datos: 13 modelos principales + all_models.py
- APIs REST: 14 endpoints principales
- Servicios: 13 servicios de logica de negocio
- CRUD: 9 modulos de acceso a datos

- Workers: 8 workers para tareas background
- Utilidades: 6 modulos de funciones auxiliares

================================================================================
2. FINALIDAD DEL PROYECTO
================================================================================

PROPOSITO PRINCIPAL:
Proporcionar una plataforma integral para la gestion operativa de servicios
de delivery, permitiendo la coordinacion eficiente entre clientes, repartidores
y operadores del servicio.

OBJETIVOS ESPECIFICOS:

a) Gestion de Ordenes:
   - Recepcion de pedidos desde multiples canales (app, web, ERP, TPV)
   - Asignacion inteligente a repartidores disponibles
   - Seguimiento en tiempo real del estado de entrega
   - Calculo automatico de tiempos estimados (SLA)
   - Estados: pending, assigned, in_progress, delivered, cancelled, failed

b) Gestion de Repartidores:
   - Registro y validacion de documentos (CNH, antecedentes)
   - Control de turnos (check-in/check-out)
   - Monitoreo de ubicacion en tiempo real
   - Sistema de gamificacion (niveles, badges, rankings)
   - Calculo de ganancias y liquidaciones
   - Estados: inactive, pending_approval, active, suspended, blocked

c) Operaciones:
   - Dashboard operativo para supervisores
   - Alertas automaticas por desvios de ruta o retrasos
   - Reasignacion dinamica de ordenes
   - Gestion de incidencias y fallas
   - Metricas de rendimiento en tiempo real

d) Analisis Financiero:
   - Calculo de costos operativos por entrega
   - Configuracion de reglas de pago flexibles
   - Liquidacion automatica diaria/semanal
   - Reportes de rentabilidad por zona/horario
   - Integracion con sistemas contables

e) Cumplimiento Legal (LGPD):
   - Auditoria completa de todas las acciones
   - Consentimiento explicito para datos personales
   - Retencion controlada de datos sensibles
   - Enmascaramiento de informacion personal
   - Derecho al olvido implementado

5. POSIBLES FALLAS DETECTADAS Y SOLUCIONADAS
================================================================================

FALLAS CRITICAS - ESTADO EN V3:

a) Inconsistencia en Modelo AuditLog y Notification:
   UBICACION: /workspace/backend/app/models/audit_log.py, notification.py
   PROBLEMA ORIGINAL: Usan Column(Integer, primary_key=True) en lugar de UUID
   ESTADO EN V3: CORREGIDO - Ambos modelos usan UUID correctamente
   IMPACTO: Inconsistencia con el resto de modelos que usan UUID
   SOLUCION APLICADA: Cambiar a UUID para mantener consistencia
   
   CODIGO ACTUAL audit_log.py:
   - id: Mapped[uuid.UUID] = mapped_column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)
   - ESTADO: CORREGIDO - Usa UUID correctamente
   
   CODIGO ACTUAL notification.py:
   - id: Mapped[uuid.UUID] = mapped_column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)
   - ESTADO: CORREGIDO - Usa UUID correctamente

b) Modelos Duplicados:
   UBICACION: /workspace/backend/app/models/all_models.py
   PROBLEMA: Existen definiciones duplicadas de modelos principales
   ESTADO EN V3: PRESENTE - Requiere decision de diseño
   IMPACTO: Confusion sobre cual version usar
   SOLUCION RECOMENDADA: 
     Opcion A: Eliminar all_models.py y usar solo modelos individuales
     Opcion B: Convertir all_models.py en modulo de exportacion (__all__)
   
   RECOMENDACION: Mantener all_models.py SOLO como punto de exportacion:
   ```python
   from app.models.user import User, UserRole
   from app.models.rider import Rider, RiderStatus, VehicleType
   # ... etc para facilitar imports
   __all__ = ['User', 'UserRole', 'Rider', ...]
   ```

c) Relaciones Cruzadas Inconsistentes:
   UBICACION: Varios archivos de modelos
   ESTADO EN V3: VERIFICADO - Las relaciones back_populates coinciden correctamente
   IMPACTO: Sin errores en tiempo de ejecucion
   SOLUCION: Todas las relaciones bidireccionales verificadas
   
   RELACIONES VERIFICADAS:
   - User <-> Rider: user_id / user (CORRECTO)
   - User <-> AuditLog: user_id / user (CORRECTO)
   - Order <-> Delivery: order_id / order (CORRECTO)
   - Rider <-> Shift: rider_id / rider (CORRECTO)
   - Rider <-> Financial: rider_id / rider (CORRECTO)
   - Rider <-> Productivity: rider_id / rider (CORRECTO)
   - Order <-> Route: order_id / order (CORRECTO)

------------------------------------------------------------------------------
MODELO 1: User (usuarios.py)
------------------------------------------------------------------------------
TABLA: users
PRIMARY KEY: UUID (uuid.uuid4)

CAMPOS:
- id: UUID (primary_key, default=uuid.uuid4)
- email: String(255) unique, indexed, not null
- hashed_password: String(255) not null
- full_name: String(255) not null
- role: Enum(UserRole) default=OPERADOR, not null
  * SUPERADMIN, GERENTE, OPERADOR, REPARTIDOR
- is_active: Boolean default=True
- phone: String(30) nullable
- created_at: DateTime(timezone=True) default=UTC now
- updated_at: DateTime(timezone=True) onupdate=UTC now
- last_login: DateTime(timezone=True) nullable

CAMPOS LGPD:
- lgpd_consent: Boolean default=False
- lgpd_consent_date: DateTime(timezone=True) nullable
- is_deleted: Boolean default=False (soft delete)

RELACIONES:
- rider_profile: relationship("Rider", back_populates="user", uselist=False)
- audit_logs: relationship("AuditLog", back_populates="user")

INDEXES:
- email (unique, indexed)

------------------------------------------------------------------------------
MODELO 2: Rider (rider.py)
------------------------------------------------------------------------------
TABLA: riders
PRIMARY KEY: UUID (uuid.uuid4)

CAMPOS:
- id: UUID (primary_key, default=uuid.uuid4)
- user_id: UUID foreign key -> users.id, unique, not null
- cnh_number: String(20) nullable (Carteira Nacional de Habilitacao)
- cnh_expiry: Date nullable
- vehicle_type: Enum(VehicleType) default=MOTO
  * MOTO, BICICLETA, CARRO, VAN
- vehicle_plate: String(10) nullable
- status: Enum(RiderStatus) default=PENDING_APPROVAL
  * INACTIVE, PENDING_APPROVAL, ACTIVE, SUSPENDED, BLOCKED
- rating: Float default=0.0 (0-5 estrellas)
- total_deliveries: Integer default=0
- level: Integer default=1 (gamificacion)
- badges: JSON default=[] (logros obtenidos)
- current_location: Geometry('POINT', 4326) nullable (PostGIS)
- created_at: DateTime(timezone=True) default=UTC now
- updated_at: DateTime(timezone=True) onupdate=UTC now

RELACIONES:
- user: relationship("User", back_populates="rider_profile")
- shifts: relationship("Shift", back_populates="rider")
- financial: relationship("Financial", back_populates="rider", uselist=False)
- productivity: relationship("Productivity", back_populates="rider")
- deliveries: relationship("Delivery", back_populates="rider")

INDEXES:
- user_id (unique)
- status (indexed)
- current_location (spatial index)

8. APIS REST IMPLEMENTADAS
================================================================================

BASE URL: /api/v1

------------------------------------------------------------------------------
AUTH ENDPOINTS (auth.py)
------------------------------------------------------------------------------
POST   /auth/login                    # Login con OAuth2PasswordRequestForm
       - Recibe: username (email), password
       - Retorna: access_token, refresh_token, user_id, role, full_name
       
POST   /auth/refresh                  # Refresh de token
       - Recibe: refresh_token
       - Retorna: Nuevo access_token y refresh_token
       
POST   /auth/register-rider           # Registro publico de repartidores
       - Recibe: email, password, full_name, phone, vehicle_type, lgpd_consent
       - Retorna: Mensaje de exito, user_id
       
GET    /auth/me                       # Obtener usuario actual
       - Headers: Authorization: Bearer <token>
       - Retorna: id, email, full_name, role, is_active

10. SISTEMAS DE AUTENTICACION Y SEGURIDAD
================================================================================

------------------------------------------------------------------------------
AUTENTICACION JWT
------------------------------------------------------------------------------
IMPLEMENTACION:
- Libreria: python-jose[cryptography]
- Algoritmo: HS256
- Secret Key: Configurada via variable de entorno SECRET_KEY

ACCESS TOKEN:
- Duracion: 30 minutos (configurable)
- Payload: {sub: user_id, exp: expiration, type: "access"}
- Uso: Autorizacion en cada request

REFRESH TOKEN:
- Duracion: 7 dias (configurable)
- Payload: {sub: user_id, exp: expiration, type: "refresh"}
- Uso: Obtener nuevo access token sin re-autenticar

FLOW DE AUTENTICACION:
1. Usuario envia credenciales (email/password)
2. Backend valida credenciales contra base de datos
3. Si valido, genera access_token y refresh_token
4. Frontend almacena tokens (localStorage/secure cookie)
5. Cada request incluye access_token en header Authorization
6. Cuando access_token expira, usar refresh_token para obtener nuevo
7. Cuando refresh_token expira, requerir login nuevamente

------------------------------------------------------------------------------
HASH DE CONTRASEÑAS
------------------------------------------------------------------------------
IMPLEMENTACION:
- Libreria: passlib[bcrypt]
- Algoritmo: bcrypt
- Rounds: 12 (default)

FUNCIONES:
- hash_password(password) -> str
- verify_password(plain_password, hashed_password) -> bool

REQUISITOS DE CONTRASEÑA:
- Minimo 8 caracteres
- Al menos una mayuscula
- Al menos un numero
- Al menos un caracter especial

------------------------------------------------------------------------------
AUTORIZACION POR ROLES
------------------------------------------------------------------------------
ROLES DISPONIBLES:
1. SUPERADMIN: Acceso completo a todo
2. GERENTE: Gestion de operaciones, aprobacion de repartidores
3. OPERADOR: Gestion diaria de ordenes y entregas
4. REPARTIDOR: Solo acceso a sus propias entregas y turnos

IMPLEMENTACION:
- Decorador require_role(*roles)
- Verificacion en cada endpoint protegido
- Roles definidos en Enum UserRole

EJEMPLO:
```python
@router.get("/admin-only")
async def admin_endpoint(current_user: User = Depends(require_role(UserRole.SUPERADMIN, UserRole.GERENTE))):
    return {"message": "Solo admins"}
```

------------------------------------------------------------------------------
RATE LIMITING
------------------------------------------------------------------------------
IMPLEMENTACION:
- Middleware personalizado: RateLimitMiddleware
- Almacenamiento: Redis
- Limite default: 60 requests por minuto por IP

CONFIGURACION:
- RATE_LIMIT_PER_MINUTE: 60 (en config.py)
- Redis key pattern: ratelimit:{ip}:{endpoint}
- Ventana deslizante de 60 segundos

RESPUESTA CUANDO EXCEDE LIMITE:
- HTTP 429 Too Many Requests
- Headers: X-RateLimit-Limit, X-RateLimit-Remaining, Retry-After

IMPLEMENTACION:
